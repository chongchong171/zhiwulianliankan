## 核心玩法规则

### 基础机制

**消除规则：**
- **3 消机制**：三张相同图案的卡牌放入卡槽后自动消除
- **堆叠布局**：卡牌多层堆叠，上层卡牌压下层卡牌
- **拾取限制**：只有最上层（未被覆盖）的卡牌可点击拾取
- **卡槽容量**：7 个格子
- **失败条件**：卡槽满 7 张牌且无法消除
- **胜利条件**：消除所有卡牌

### 卡槽机制

```
卡槽容量：7 格

状态示例：
[🐑][🐑][🐒][🦊][🦊][🐼][ ]  ← 6 张，还可继续
[🐑][🐒][🦊][🐼][🐮][🐔][🐶]  ← 7 张不同，失败！
[🐑][🐑][🐑]                  ← 3 张相同，自动消除
```

**关键规则：**
1. 卡牌进入卡槽后，立即检查是否有 3 张相同
2. 若有 3 张相同，立即消除并腾出空间
3. 第 7 张牌放入后若无法消除，游戏结束
4. 消除后卡槽内卡牌自动向左对齐

### 堆叠层级规则

**层级覆盖逻辑：**
- 第一层（最底层）的卡牌可能被第二层、第三层等多层覆盖
- 判断某张牌是否可点击：检查其上方所有可能覆盖位置是否有牌
- 覆盖规律（以第一层 (2,4) 位置为例）：
  - 需检查第二层的 4 个位置：(2,4)、(3,4)、(2,3)、(3,3)
  - 任一位置有牌，则该牌不可拾取

**层级设计：**
- 第一关：约 3 层，教学关，不可能输
- 第二关：10+ 层，地狱难度，通关率约 0.1%

## 关卡设计

### 第一关（教学关）

**特点：**
- 牌数少（约 20-30 张）
- 层级少（2-3 层）
- 牌型种类少（3-5 种）
- **设计目标**：让玩家熟悉规则，建立"很简单"的错觉
- **通关率**：接近 100%

### 第二关（核心关）

**特点：**
- 牌数多（约 100-200 张）
- 层级多（10+ 层）
- 牌型种类多（10-15 种）
- **设计目标**：极高难度，迫使玩家看广告
- **通关率**：约 0.1%（千分之一）

**难度设计哲学：**
> "第一关上幼儿园，第二关去考研"

**关键设计：**
1. **反常规难度曲线**：不是渐进式难度，而是断崖式提升
2. **随机生成**：每次重玩牌型分布不同
3. **死局设计**：部分牌局天生无解，必须使用道具
4. **运气成分**：通关需要运气 + 技巧

### 牌型生成算法

**可通关的随机生成逻辑：**

```
1. 确定关卡总牌数 N（必须是 3 的倍数）
2. 确定牌型种类数 M（如 10 种）
3. 从 M 种中随机选取 K 种用于本局
4. 分组生成：
   - 每次随机选 1 种牌型
   - 生成 3 张该类型的牌 [A,A,A]
   - 重复直到总数达到 N
5. 打乱顺序：将生成的牌列表随机打乱
6. 分配到层级：按关卡布局分配到各层位置
```

**关键点：**
- 保证每种牌都是 3 的倍数（理论上可全部消除）
- 但堆叠顺序随机，可能导致关键牌被压在深处
- 这就是"理论上可通关，实际上极难"的设计

```javascript
// 卡牌对象
{
  id: "card_001",
```

### 核心逻辑

**1. 卡牌可点击判断：**

```javascript
function isCardClickable(card) {
  if (card.layer === maxLayer) return true;  // 最上层
  
  // 检查上一层的 4 个覆盖位置
  const coverPositions = [
    {row: card.row, col: card.col},
    {row: card.row + 1, col: card.col},
    {row: card.row, col: card.col - 1},
    {row: card.row + 1, col: card.col - 1}
  ];
  
  for (let pos of coverPositions) {
    if (hasCardAtLayer(pos, card.layer + 1)) {
      return false;  // 被覆盖，不可点击
    }
  }
  return true;
}
```

**2. 消除检测：**

```javascript
function checkElimination(slot) {
  // 统计每种牌型的数量
  const typeCount = {};
  for (let card of slot) {
    if (card) {
      typeCount[card.type] = (typeCount[card.type] || 0) + 1;
    }
  }
  
  // 检查是否有 3 张相同
  for (let type in typeCount) {
    if (typeCount[type] === 3) {
      // 消除这 3 张牌
      eliminateCards(type);
      return true;
    }
  }
  return false;
}
```

**3. 失败检测：**

```javascript
function checkFailure(slot) {
  // 卡槽满 7 张且无相同 3 张
  const filledCount = slot.filter(c => c !== null).length;
  if (filledCount === 7) {
    if (!checkElimination(slot)) {
      return true;  // 失败
    }
  }
  return false;
}
```

### 道具系统

| 道具 | 功能 | 获取方式 |
|------|------|----------|
| **移出** | 移除卡槽前 3 张牌 | 看广告 / 分享 |
| **洗牌** | 打乱所有未消除牌的顺序 | 看广告 / 分享 |
| **复活** | 失败后清空卡槽继续 | 看广告 / 分享 |
| **撤回** | 撤销上一步操作 | 看广告 / 分享 |

**道具使用策略：**
- **洗牌**：核心道具，当关键牌被压住时使用
- **移出**：卡槽快满时救急
- **复活**：最后关头使用
- **最佳时机**：卡槽有 5 张以上且局面僵化时

## 用户心理分析

### 利用的人性弱点

| 心理 | 应用方式 |
|------|----------|
| **好奇** | "莫名其妙火了，我也要试试" |
| **虚荣** | 省份排名，为家乡争光 |
| **傲慢** | "别人过不了，我肯定能过" |
| **社交** | 话题讨论，分享炫耀 |
| **冲突** | 游戏机制引发心理冲突 |
| **赌徒心理** | "再试最后一局" |
| **损失厌恶** | 已投入时间，不愿放弃 |

### 心流理论的反向应用

**传统游戏设计：**
- 难度渐进
- 即时正反馈
- 让用户进入心流状态

**羊了个羊设计：**
- 难度断崖
- 负反馈为主（失败）
- 让用户在心流与非心流间徘徊
- 产生焦虑 + 愤怒情绪

**效果：**
- 用户质疑自己智商 → 继续尝试
- 用户质疑游戏有问题 → 引发讨论
- 形成话题传播

### 沉没成本效应

```
用户心理循环：
1. 玩第一关 → "很简单"
2. 玩第二关 → "不可能吧"
3. 失败 → "再来一局"
4. 连续失败 → "已经玩了这么久，不能放弃"
5. 看广告 → "最后一局"
6. 继续失败 → 循环...
```

## 复刻建议

### 技术栈推荐

**前端（小程序）：**
- 微信小程序原生开发
- 或使用 uni-app 跨平台框架
- 游戏引擎：Cocos Creator / LayaAir

**后端：**
- 用户数据存储（通关记录、排名）
- 地域统计
- 防作弊验证

**服务器：**
- 初期：微信云开发（低成本）
- 后期：自建服务器（应对高并发）

### 核心功能优先级

**P0（必须）：**
1. 堆叠卡牌布局
2. 卡牌拾取逻辑
3. 卡槽消除机制
4. 失败/胜利判定
5. 基础关卡（2 关）

**P1（重要）：**
1. 道具系统
2. 广告接入
3. 地域排名
4. 分享功能

**P2（可选）：**
1. 战队系统
2. 每日挑战
3. 皮肤系统
4. 成就系统

### 差异化建议

⚠️ **注意：** 直接抄袭有法律风险，建议做差异化创新

**可改进方向：**

1. **主题差异化**
   - 植物主题（结合你的 AI 植物管家项目）
   - 动物主题
   - 食物主题
   - 文化 IP 主题

2. **玩法创新**
   - 加入特殊牌（炸弹、万能牌）
   - 限时模式
   - 连击奖励
   - 技能系统

3. **难度优化**
   - 多关卡渐进难度（不要学羊了个羊的断崖式）
   - 可调节难度
   - 新手引导优化

4. **变现优化**
   - 加入内购（去广告、无限道具）
   - 会员系统
   -  Battle Pass

⚠️ **法律风险：**
- 玩法抄袭边界模糊，但美术资源、代码不能直接复制
- 建议参考玩法，自己实现，用原创美术

⚠️ **市场风险：**
- 小游戏生命周期短（1-3 个月）
- 需要快速迭代和持续运营
- 爆款可遇不可求

⚠️ **道德风险：**
- 避免过度利用人性弱点
- 平衡盈利与用户体验
- 考虑长期品牌建设
